PRD030 - Army Snapshot Fields

Draft do grillowania bazy Offline Mode. Snapshot oznacza zapis armii w konkretnym momencie runu albo saved-army rosteru.

Decyzja Kierunkowa

Baza przechowuje runtime state. Nie przechowuje pełnych katalogów jednostek, skilli, opisów, ikon, kosztów ani tierów jako authored content.

Snapshot armii zapisuje to, co gracz faktycznie ma: jakie stacki, ile sztuk, jakie skille są przypisane, na którym node runu powstał ten stan oraz kiedy został zapisany.

Keep - trzymać w V1 Maybe - decyzja produktowa Cut - raczej wyciąć

Table: army_snapshots

Field Decision Po co Uwagi
snapshot_id Keep Unikalne id snapshotu. Potrzebne do łączenia z runem, saved army, battle, reward i shop.
account_id Keep Właściciel snapshotu. Lokalnie na razie może być jedno konto, ale online każdy gracz będzie miał własne id. Warto mieć ten sam kształt danych od początku.
run_id Keep Łączy snapshot z runem. Może być puste dla snapshotu saved army importowanego spoza runu.
saved_army_id Keep Łączy snapshot z zapisaną armią. Może być puste dla snapshotów w trakcie runu.
node_id Keep Node runu, przy którym powstał snapshot. Zastępuje wcześniejsze kind / source. Z kontekstu node można odczytać, czy chodziło o battle, reward, shop, final itd. Może być puste dla snapshotów spoza runu, np. import/dev seed.
created_at_utc Keep Czas utworzenia. Przydatne dla save/load, debugowania i historii.
kind Cut Typ snapshotu. Niepotrzebne, bo node_id daje kontekst runu.
source Cut Źródło snapshotu. Niepotrzebne, bo dubluje kind i w praktyce zastępuje je node_id.
note Cut Dowolna notatka debugowa. Niepotrzebne w V1. Lepiej użyć osobnej tabeli debug events, jeśli będzie potrzebna.

Table: army_snapshot_stacks

Field Decision Po co Uwagi
snapshot_stack_id Keep Unikalne id stacka w tym snapshotcie. Potrzebne, żeby przypiąć skille do konkretnego stacka.
snapshot_id Keep Łączy stack ze snapshotem. Każdy stack należy do jednego snapshotu.
stack_id Cut Osobne id stacka wewnątrz runu. Niepotrzebne, skoro istnieje snapshot_stack_id. Porównania i targetowanie powinny opierać się na snapshot stack id albo formation slot, zależnie od flow.
unit_id Keep Jaka jednostka jest w stacku. Odwołuje się do authored unit catalog. Nie zapisuje pełnej definicji jednostki.
amount Keep Ile jednostek jest w stacku. Główna wartość saved army i run army.
lost Cut Ile jednostek jest straconych/rannych w tym stanie. Niepotrzebne, jeśli straty liczymy przez porównanie snapshotu przed i po walce.
level Cut Poziom stacka, jeśli stacki mają progresję. Nie wchodzi do V1 snapshotu.
experience Cut XP stacka, jeśli stacki mają progresję. Nie wchodzi do V1 snapshotu.
formation_slot Keep Pozycja stacka w rosterze/układzie armii. Potrzebne do przyszłego układania armii i stabilnej kolejności wyświetlania.
sort_order Cut Stabilna kolejność wyświetlania. Niepotrzebne, bo sortowanie idzie po formation_slot.
temporary_power Cut Tymczasowa moc/buff stacka. Raczej nie mieszać z armią. Buff/debuff powinien mieć osobny model, jeśli kiedyś będzie zapisywany.

Table: army_snapshot_stack_skills

Field Decision Po co Uwagi
snapshot_stack_skill_id Keep Unikalne id rekordu skilla. Proste primary key.
snapshot_stack_id Keep Łączy skill ze stackiem. Skill unlock jest per unit stack, nie army-wide.
skill_id Keep Jaki skill jest zapisany. To ten sam string, którego używa obecny skill system.
acquired_at_run_node_id Keep Na którym node runu skill został zdobyty. Zastępuje source. Jeśli skill jest przypisany w snapshotcie, znaczy że jest dostępny dla tego stacka. Metaprogress unlock należy do osobnej tabeli account progress, nie do snapshotu.
unlocked Cut Czy skill jest odblokowany. Niepotrzebne. Jeśli skill jest przypisany do stacka w snapshotcie, jest dostępny. Globalne odblokowanie możliwości pojawienia się skilla w runie należy do account/metaprogress DB.
source Cut Skąd stack dostał skill. Niepotrzebne, używamy acquired_at_run_node_id.

Fields To Avoid In Snapshot

Field Decision Dlaczego nie
display_name Cut Pochodzi z unit catalog.
tier Cut Pochodzi z unit catalog, chyba że kiedyś tier stanie się runtime upgrade state.
unit_cost Cut Pochodzi z unit catalog/balansu.
combat_value Cut Powinno być wyliczane z aktualnego katalogu i amount, jeśli nie chcemy historycznych kopii wartości.
skill_description Cut Pochodzi z skill catalog/XML/presentation data.
skill_icon Cut Pochodzi z Resources/presentation data.
