PRD030 Offline Database Draft

High-Level Grill Questions

Te pytania ustalają model całej bazy przed zejściem w szczegóły pól.

Database table style

Dokument używa formatu: Field, Type, Nullable, Default value, Description, Related to / FK, Status.

Globalna zasada DB: korzystamy z foreign keys, nie robimy hard delete, a dezaktywacja rekordu oznacza is_active = 0. Statusy biznesowe są integer enum ids z DB Enums.cs.

# Pytanie Rekomendacja
1 Czy armia w bazie jest zawsze zapisywana jako nowy snapshot po każdej ważnej zmianie, zamiast mutować stary snapshot? Tak. Snapshoty powinny być historyczne/niemutowalne. Run wskazuje aktualny snapshot przez current_army_snapshot_id.
2 Czy database V1 tworzy pełny schema od razu, a ekrany podpinają się później taskami? Tak. Pełny schema V1 ogranicza drift między Reward, Shop, Summary i Saved Armies.
3 Czy baza zapisuje tylko runtime state, bez authored unit/skill/reward/shop catalogs? Tak. DB zapisuje id i stan. Katalogi zostają authored data.
4 Czy istnieje jedno lokalne konto w Offline Mode, ale z account_id gotowym pod przyszłe online ids? Tak. Lokalnie default account, ale w przyszlosci moze byc wiecej kont jako osobne save/profile. Wszystkie glowne rekordy maja account_id.
5 Czy node_id zastępuje kind/source na snapshotach? Tak. Dla final snapshot moze to byc final node. Snapshot spoza runu powinien pochodzic z jawnego seed/setup node albo osobnej operacji seedujacej.
6 Czy saved army jest zawsze snapshotem, a overwrite tworzy nowe saved_army_id? Tak. Saved army ma status aktywna/nieaktywna. Overwrite ustawia stara jako inactive i zapisuje nowa z nowym id.
7 Czy battle/reward/shop zapisują event records oraz nowy snapshot armii? Tak. Calosc idzie przez jedna tabele run_events. Reward, purchase i battle maja swoje rekordy szczegolowe z referencja do eventu.
8 Czy metaprogress unlock decyduje, co może pojawić się w runie, a snapshot zapisuje tylko to, co faktycznie jest w armii? Tak. Skill unlock metaprogress nie powinien siedzieć w snapshotach armii.
9 Czy Route Nodes są głównym kontekstem przepływu runu? Tak. Route nodes sa seedowane jako cala mapa runu od razu po starcie runu. Potem zmienia sie tylko ich runtime state.
10 Czy legacy Arena/custom armies sa calkowicie poza baza PRD030? Tak. Rezygnujemy z import/helper w PRD030. Jesli potrzeba danych testowych, tworzymy na sztywno seed snapshot w nowej bazie.
11 Czy typy eventow i statusow maja byc zamknieta lista kontrolowana przez kod? Tak. W bazie trzymamy enum id jako integer, a mapowanie nazw siedzi w jednym pliku DB Enums.cs.
12 Czy enum ids maja byc stale i recznie przypisane? Tak. Nie moga wynikac z kolejnosci enumow w C#. Te same ids beda pozniej przydatne do tlumaczen, UI labels i stabilnego persistence contract.
13 Czy node_id ma byc calkowicie unikalnym, zwyklym integer id z jednej tabeli node? Tak. Jedna kanoniczna tabela route_nodes, a node_id to plain integer primary key. Snapshoty, eventy, battle, reward i shop referencja ten int.
14 Czy lokalne primary key w SQLite maja byc zwyklymi integer ids zamiast string/Guid? Tak. Lokalne rekordy DB uzywaja integer primary key i integer FK. Tekstowe zostaja tylko authored/catalog ids, np. unit_id, skill_id, template_id, encounter_id. Przyszle online/sync id dodajemy jako osobne opcjonalne pole.
15 Czy uzywamy foreign keys i soft delete zamiast kasowania rekordow? Tak. SQLite ma pracowac z PRAGMA foreign_keys = ON. Nie robimy hard delete w normalnym flow; rekordy dezaktywujemy przez is_active = 0. Nie stosujemy agresywnego cascade delete dla historii runu.
